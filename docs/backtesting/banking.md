# Banking cores — FLEXCUBE, Temenos, Nucleus, Murex (triage: all gated)

Requested targets: Oracle FLEXCUBE, Temenos Transact/T24, Nucleus FinnOne,
Murex MX.3. Verdict after live-fetch research: **none is backtestable** under
this project's bar (a real spec fetchable without a tenant/partner login AND a
mature reference MCP server to compare against). Zero banking-core MCP servers
exist publicly — unsurprising, since every one of these products is deployed
on-prem or single-tenant behind bank firewalls.

## Oracle FLEXCUBE — PARTIAL (closest to backtestable, still short)

The one genuinely public artifact: docs.oracle.com hosts a **ServiceXML
documentation zip (~49MB) containing 10,216 XSD files** describing FLEXCUBE's
gateway message schemas. What it does *not* contain is any service contract —
no WSDL, no OpenAPI, no operation list. FLEXCUBE's actual interfaces are:

- **Gateway SOAP services** — WSDLs are generated by and served from an
  installed instance (`/FCUBSAccService/FCUBSAccService?WSDL` style); not
  published anywhere public.
- **REST swagger** — likewise generated per instance by the Open Development
  Tool; the swagger JSON is a runtime artifact of a licensed installation.

So Anvil could ingest the XSD *types* but would have no operations to compile
— and there is no reference MCP to diff against. If a FLEXCUBE instance WSDL
ever lands in hand, the multi-file WSDL mechanism built for Travelport
(`gds.md`) is exactly the machinery it would need (instance WSDLs import the
same kind of XSD trees).

## Temenos Transact — GATED

api.temenos.com has a real, well-organized API catalog (OpenAPI-based,
Transact/Infinity), but every spec download sits behind a Temenos community
login. No public GitHub mirror of the specs, no MCP server.

## Nucleus FinnOne — GATED

No public developer portal at all. Nucleus Software publishes marketing
material for FinnOne Neo's "open APIs" but no spec artifact of any format is
reachable without a customer relationship. No MCP server.

## Murex MX.3 — GATED

MX.3's integration surface (mxplus, MxML exchange) is documented exclusively
inside client/partner portals under NDA. Nothing fetchable, no MCP server.

## What this run contributed anyway

The banking attempt is why the **GDS systems** became the WSDL proving ground:
the user's follow-up ("try GDS systems if they're WSDL based") led to
Travelport's fully public 158-WSDL tree, which drove the multi-file WSDL/XSD
resolution mechanism — the same mechanism a FLEXCUBE instance WSDL would need
on day one. See `gds.md`.
