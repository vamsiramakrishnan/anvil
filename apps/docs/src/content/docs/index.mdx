## Any spec in. Every surface out.

Enterprises don't lack APIs — they lack **agent-safe** ones. The payments API
knows a refund moves money; the WSDL knows the transfer can't be undone. But
none of that knowledge survives the trip into an agent's toolbox when wrappers
are written by hand. Anvil compiles it in.

<PipelineDiagram />

Everything on the right is a **projection** of the model in the middle. Nobody
hand-writes the CLI, the MCP tool descriptions, or the skill — so nobody can
forget that the refund needs an idempotency key on one surface but not another.

## The problem isn't writing wrappers. It's keeping them honest.

Any team can wrap an API for one demo. The failure mode arrives six months
later, when the CLI wrapper retries a payment the MCP wrapper wouldn't, and the
runbook says a third thing.

## Where enterprises point it first

<Tabs>
  <TabItem label="Payments & billing">

A refund is a **financial, irreversible mutation**. Anvil derives that from the
spec, and every surface enforces it: the call refuses to run without
`confirm: true` and an idempotency key, and it is never auto-retried.

```bash
anvil compile stripe-payments.yaml --manifest payments.anvil.yaml \
  --human-approval unsafe --out generated/payments

anvil inspect generated/payments
# payments.refunds.create   mutation · financial · irreversible
#   confirmation: required (human approval)   idempotency: required
```

With `--human-approval unsafe`, the generated hooks escalate the refund to a
**person** — a model-supplied `confirm: true` can't clear the gate.

  </TabItem>
  <TabItem label="Legacy core systems">

The core banking system speaks SOAP, and it's not getting rewritten this
decade. Anvil compiles multi-file WSDL directly — the twenty-year-old spec
becomes typed, gated agent tools with the same safety contract as the newest
OpenAPI service.

```bash
anvil compile core-banking.wsdl --manifest banking.anvil.yaml --out generated/banking
```

No modernization project as a prerequisite. The spec you have is the input.

  </TabItem>
  <TabItem label="HR & PII">

For an HCM system, reads are useful and mostly harmless — compensation changes
are neither. Approve the reads broadly, gate the writes individually, and mark
the sensitive mutations `human_approval: true` in the manifest so no agent can
self-approve a salary change.

```yaml
# hcm.anvil.yaml
operations:
  updateCompensation:
    confirmation: { required: true, human_approval: true }
```

  </TabItem>
  <TabItem label="Customer messaging">

An outbound SMS can't be unsent, and a timeout doesn't mean it wasn't
delivered. Anvil marks sends **not retry-safe** — the runtime never auto-retries
them — and requires an idempotency key, so a nervous agent re-issuing the call
is a duplicate-suppressed no-op instead of a second text to your customer.

  </TabItem>
</Tabs>

## Three rings, one contract

Hooks make enforcement cheap and early; the runtime makes it inescapable. Every
check lives in the innermost ring that can enforce it — the outer rings buy
speed and human escalation, never correctness.

<RingsDiagram />

## The loop

```bash
# 1. Compile a spec into a full bundle (CLI + MCP + skill + hooks)
anvil compile <spec> --manifest <manifest> --out <dir>

# 2. Read every operation's effect, risk, and idempotency
anvil inspect <dir>

# 3. Expose an operation only after reading its risk
anvil approve <dir> <operation-id>

# 4. Package the skill and deploy the server
anvil package skill <dir>
```

Non-idempotent mutations land as `review_required` — not exposed until a human
approves them. That refusal is the feature, not the bug.
