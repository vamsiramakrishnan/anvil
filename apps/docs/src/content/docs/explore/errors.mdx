Anvil never leaks raw upstream chaos as the primary interface: every failure —
from any generated CLI or MCP tool — becomes exactly one structured envelope
with a code from the closed taxonomy below.

```json
{ "error": { "code": "rate_limited", "message": "...", "retryable": true, "safe_to_retry": true, "operation": "...", "trace_id": "..." } }
```

The code list on this page is imported from `apps/docs/src/data/errors.json`,
which is produced by an actual compile (`node scripts/gen-docs-data.mjs`) and
**test-enforced against the runtime taxonomy** by
`packages/generators/src/docs-data.test.ts` — the page physically cannot list a
code the runtime no longer returns. The recovery prose mirrors the
`reference/errors.md` file that ships inside every generated skill (see
`errorsRef()` in `packages/generators/src/skill.ts`), so what you read here is
what deployed agents read.

export const codeText = (s) =>
  s.split(/(`[^`]+`|\*\*[^*]+\*\*)/g).map((part) =>
    part.startsWith("`") ? (
      <code>{part.slice(1, -1)}</code>
    ) : part.startsWith("**") ? (
      <strong>{part.slice(2, -2)}</strong>
    ) : (
      part
    ),
  );

Two fields in the envelope carry the retry contract: `retryable` says whether
the *class* of failure is transient, and `safe_to_retry` says whether **this
operation** may actually be retried given its idempotency posture. An agent
should only ever act on `safe_to_retry` — the runtime has already retried
whatever it safely could.
